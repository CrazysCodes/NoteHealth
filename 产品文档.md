# NoteHealth 产品文档

## 1. 产品概述

**NoteHealth** 是一个浏览器端、本地优先的 Markdown 知识库健康检查工具。

它面向所有长期使用 Markdown 写笔记、写研究记录、写技术文档或维护个人知识库的人。用户通过浏览器选择本地 Markdown 文件夹，NoteHealth 在前端内存中读取并分析 `.md` 文件，生成知识库健康报告和整理建议。

NoteHealth 不是 Obsidian 插件，不是 AI 写作工具，也不是团队知识库 SaaS。第一版目标是做成一个小而可信、开箱即用的本地分析工具。

## 2. 产品定位

一句话定位：

> A local-first health checker for your Markdown knowledge base.

中文解释：

> 一个本地优先的 Markdown 知识库体检工具，帮助用户发现笔记库中的孤岛笔记、断链、陈旧内容、标签混乱、结构冗杂和整理优先级。

## 3. 目标用户

- Typora 用户
- Obsidian 用户
- VS Code / Markdown 笔记用户
- 使用本地文件夹管理研究笔记、技术笔记、读书笔记的人
- 使用 Git 仓库维护个人文档或知识库的人
- 长期写作、研究、学习，但笔记数量逐渐失控的个人用户

## 4. 核心痛点

用户长期积累 Markdown 笔记后，常见问题包括：

- 笔记越来越多，但缺少结构
- 很多笔记写完后再也找不到
- 笔记之间缺少链接，形成信息孤岛
- 标签命名混乱，例如 `#ai`、`#AI`、`#llm`
- 部分笔记长期未更新，可能已经过期
- 目录层级越来越深，查找成本变高
- 某些主题已经有大量笔记，但缺少索引页
- 用户不知道应该从哪里开始整理

NoteHealth 的价值不是替用户写笔记，而是帮助用户看清知识库当前状态，并给出可执行的整理路线。

## 5. MVP 范围

### 5.1 MVP 核心交互

用户打开 Web 页面后：

1. 点击 `Select Markdown folder`
2. 浏览器请求用户选择本地文件夹
3. 前端读取文件夹中的 `.md` 文件
4. 在浏览器内存中解析和分析
5. 展示健康报告
6. 生成整理建议
7. 用户可以复制 checklist 或导出 Markdown 格式报告

### 5.2 MVP 必须支持

- 浏览器端选择本地文件夹
- 只读取 `.md` 文件
- 忽略常见无关目录：
  - `.git`
  - `node_modules`
  - `.next`
  - `dist`
  - `build`
  - `.obsidian` 可先忽略配置文件，但不排除 vault 内容
- 解析 Markdown 文件基础信息：
  - 文件路径
  - 文件内容
  - 修改时间
  - 标题
  - 字数
  - frontmatter tags
  - inline tags，例如 `#ai`
  - wiki links，例如 `[[note-name]]`
  - Markdown links，例如 `[title](../path/file.md)`
- 生成健康分数
- 生成问题列表
- 每个问题都要有 evidence
- 生成 Action Plan
- 支持导出 Markdown 报告

### 5.3 MVP 不做

- 不做登录
- 不做云同步
- 不上传用户笔记
- 不做团队协作
- 不做数据库
- 不做 AI 聊天入口
- 不做 Obsidian 插件
- 不做桌面客户端
- 不做复杂知识图谱编辑器
- 不做自动修改用户文件

## 6. 隐私原则

NoteHealth 的第一版必须清楚表达：

- Runs locally in your browser
- Markdown content never leaves your device
- No account required
- Works with plain `.md` folders

产品实现上，MVP 不应把用户 Markdown 内容上传到任何服务器。所有分析应在浏览器内存中完成。

## 7. 核心指标体系

NoteHealth 的健康报告由一个总分和五个分项构成。

### 7.1 总分

`Knowledge Health Score`

范围：`0 - 100`

含义：衡量当前 Markdown 知识库的整体可维护性、可导航性和整理状态。

### 7.2 分项得分

1. **Connectivity**
   - 衡量笔记之间的链接健康度
   - 关注孤岛笔记、入链、出链、断链
2. **Freshness**
   - 衡量笔记是否长期无人维护
   - 关注超过 180 天未更新的笔记
3. **Structure**
   - 衡量目录层级和文件组织是否清晰
   - 关注目录过深、文件过于分散的问题
4. **Tags**
   - 衡量标签是否一致、可用
   - 关注大小写重复、近义标签、标签膨胀
5. **Depth**
   - 衡量笔记内容是否过薄
   - 关注字数过少、只有标题或片段化内容的笔记

## 8. MVP 诊断规则

### 8.1 Orphan Notes

定义：

- 没有入链
- 没有出链

示例 evidence：

> 42 notes have no incoming links and no outgoing links.

### 8.2 Broken Links

定义：

- wiki link 或 Markdown link 指向不存在的 Markdown 文件

示例 evidence：

> 17 links point to missing files, including `projects/llm-eval.md`.

### 8.3 Stale Notes

定义：

- 超过 180 天未更新

示例 evidence：

> 29 notes have not been updated in more than 180 days.

### 8.4 Thin Notes

定义：

- 内容少于 120 词，或正文非常短

示例 evidence：

> 14 notes contain fewer than 120 words.

### 8.5 Duplicate Tags

定义：

- 标签大小写不同但语义相同
- 标签命名相近
- 标签可能可以合并

示例 evidence：

> `#ai`, `#AI`, `#artificial-intelligence`, and `#llm` overlap heavily.

### 8.6 Missing Index

定义：

- 某个主题下存在多篇相关笔记，但没有 index / overview / map of content 类笔记

示例 evidence：

> 18 notes mention “retrieval” but no `retrieval-index.md` exists.

### 8.7 Deep Folders

定义：

- 文件路径目录层级过深，例如超过 4 层

示例 evidence：

> 18% of notes are nested deeper than 4 folder levels.

## 9. 核心页面

### 9.1 Folder Intake

目的：

让用户选择本地 Markdown 文件夹，并立即理解隐私边界。

核心元素：

- 品牌：NoteHealth
- 主标题：Check the health of your Markdown knowledge base.
- 副标题：Select a local folder of Markdown notes. NoteHealth scans links, tags, freshness, and structure directly in your browser.
- 主按钮：Select Markdown folder
- 次按钮：Try sample vault
- 隐私说明：
  - Runs locally in your browser
  - Notes are never uploaded
  - No account required
- 兼容性说明：
  - Plain Markdown
  - Typora folders
  - Obsidian vaults
  - Git-backed docs

### 9.2 Scanning

目的：

展示本地文件夹正在被读取和分析。

核心元素：

- 标题：Reading your notes locally.
- 进度条
- 当前统计：
  - Markdown files
  - Internal links
  - Tags
  - Folders
- 扫描步骤：
  - Reading Markdown files
  - Parsing headings and tags
  - Resolving wiki links and Markdown links
  - Detecting orphan notes
  - Calculating health score
- 扫描日志
- 隐私提醒：Analysis stays in this browser session.

### 9.3 Health Report

目的：

给出知识库结构诊断。

核心元素：

- 总分：72 / 100
- 总结：Useful, but hard to navigate at scale.
- 分项得分：
  - Connectivity
  - Freshness
  - Structure
  - Tags
  - Depth
- 关键指标：
  - Orphan notes
  - Broken links
  - Stale notes
  - Thin notes
  - Duplicate tag groups
- 问题卡片：
  - 优先级
  - 问题标题
  - Evidence
  - 影响说明
- 主按钮：View cleanup plan

### 9.4 Action Plan

目的：

给用户一条具体、可执行的整理路线。

核心元素：

- 标题：Start with the changes that improve navigation fastest.
- 优先级整理计划：
  1. Connect orphan notes
  2. Fix broken links
  3. Normalize duplicate tags
  4. Create topic index pages
  5. Review stale notes
- 每条建议包含：
  - 行动名称
  - 为什么优先
  - 涉及的笔记数量
  - 示例文件路径
- 导出动作：
  - Export health report
  - Copy cleanup checklist
- Markdown 报告预览

## 10. 视觉方向

界面语言：英文。

整体风格：

- 中庸
- 克制
- 清爽
- 专业
- 可信
- 不过度开发者 dashboard
- 不做装饰性知识花园

视觉要求：

- 背景：off-white / light gray
- 主文字：深墨色
- 健康状态：绿色
- 待关注状态：琥珀色
- 高风险或断链：红色
- 字体：Inter 或系统 sans-serif
- 文件路径、扫描日志、evidence：monospace
- 卡片：1px 边框
- 圆角：8-12px
- 阴影：克制或无阴影
- 不使用花哨渐变、漂浮装饰球、大型 SaaS hero

## 11. 技术方向

MVP 推荐技术栈：

- Next.js
- React
- TypeScript
- Tailwind CSS
- 纯前端分析
- 不需要后端
- 不需要数据库
- 不需要 AI API

核心架构原则：

文件读取和分析逻辑必须分离。

标准输入：

```ts
type KnowledgeFile = {
  path: string;
  content: string;
  modifiedAt?: number;
};
```

标准输出：

```ts
type HealthReport = {
  score: number;
  categories: {
    connectivity: number;
    freshness: number;
    structure: number;
    tags: number;
    depth: number;
  };
  metrics: Record<string, number>;
  issues: Issue[];
  actions: ActionItem[];
};
```

推荐核心模块：

```ts
readMarkdownFolder();
parseMarkdownFiles(files);
analyzeKnowledgeBase(parsedFiles);
generateHealthReportMarkdown(report);
```

这样未来可以迁移到：

- Tauri 客户端
- CLI
- Obsidian 插件
- VS Code 插件

## 12. 后续规划

### V1：浏览器本地 MVP

目标：

验证用户是否需要一个本地 Markdown 知识库健康检查工具。

能力：

- 浏览器选择文件夹
- 本地读取 Markdown
- 规则分析
- 健康报告
- 整理建议
- Markdown 报告导出

### V1.1：体验增强

可能功能：

- 拖拽文件夹或 zip
- 示例 vault
- 更好的错误提示
- 支持更准确的 wiki link 解析
- 更详细的问题过滤和排序
- 支持点击 issue 查看相关文件列表

### V1.2：报告增强

可能功能：

- Markdown 报告模板优化
- 可配置 stale 阈值
- 可配置 thin note 阈值
- 可导出 JSON
- 可导出 CSV
- 可复制 Obsidian checklist

### V2：分析能力增强

可能功能：

- 更好的主题聚类
- 标签相似度识别
- 主题 index 缺失检测
- 文件命名一致性检查
- 周期性健康趋势对比
- 多次扫描结果比较

### V3：多外壳支持

可能形态：

- Tauri 桌面客户端
- CLI
- Obsidian 插件
- VS Code 插件

原则：

核心分析器保持独立，外壳只负责文件读取和结果展示。

### V4：可选 AI 整理建议

前提：

必须保证无 AI API key 时产品仍可完整运行。

可能功能：

- AI 总结整理计划
- AI 推荐主题索引结构
- AI 识别相似笔记
- AI 生成更自然的 cleanup report

边界：

- AI 不能成为 MVP 的核心依赖
- 必须明确用户内容是否会发送到模型
- 必须提供本地规则分析模式

## 13. 成功标准

MVP 成功的判断：

- 用户能在 1 分钟内完成首次扫描
- 用户能明确理解文件不会上传
- 用户能看懂知识库当前主要问题
- 用户能获得一份可执行的整理清单
- 用户愿意对自己的真实笔记库运行一次
- 用户愿意导出报告或 checklist 后继续整理
